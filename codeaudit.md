# Code Audit: Sveltry

> Read-only audit. Date: 2026-06-16. This report is self-contained: every finding cites exact locations and how to verify a fix, so an agent with only this file and the code can act on it. No source files were modified; the only file produced is this one.

## Snapshot

- **Project**: Sveltry, a self-hosted, Sentry-wire-compatible error tracker (open source, Apache-2.0, single-author `aihxp`).
- **State**: commit `0a19fa2`, branch `main`, working tree clean. Released version v0.9.0 (pre-1.0).
- **Languages**: TypeScript, Svelte 5 (`.svelte`), SQL (Postgres init only), Shell.
- **Frameworks**: SvelteKit 2 + Svelte 5 (runes); self-hosted open-source Convex (HTTP actions, queries, mutations, internal actions, crons); Tailwind v4 + shadcn-svelte; Better Auth on Convex (`@convex-dev/better-auth`, RS256 JWT); convex-svelte live queries; the Sentry wire protocol (a compatible subset).
- **Package manager / build**: Bun 1.3.11 workspaces (isolated linker, text `bun.lock`, dependency catalogs). All build/dev/test/check fan out via `bun run --filter '*'`.
- **Size**: 179 source files, ~23,162 LOC. By area: backend/Convex 44 files (~10.9k), dashboard 67 files (~7.1k), protocol 47 files (~4.2k), types 3 files (~0.6k), sdk 5 files (~0.3k).
- **Entry points**: `apps/backend/convex/http.ts` (route table: Better Auth, ingest `POST /api/<id>/envelope|store/`, public API `/api/v1/`, `POST /artifacts/upload`, `/deploys`, `/releases/commits`, `/healthz`); `ingest.ts` (hot path); `publicApi.ts` (Bearer v1 REST); `sourcemaps.ts` (upload + symbolication); `crons.ts` (9 scheduled jobs); `alerts.ts` (egress dispatch); the SvelteKit `(app)` route group.
- **Evident maturity**: production-aspiring, not yet production-proven. Far beyond a prototype (RBAC, per-mutation tenant isolation, audit log, SSRF-guarded egress, ingest-time PII scrubbing, rate limit + quota + spike protection, source-map symbolication, replays/profiles/sessions/monitors, thorough self-hosting docs, honest parity matrix), but the entire Convex backend and dashboard have zero automated tests.
- **Audit coverage**: the backend (Convex) and protocol packages were read close to exhaustively on the security, error-handling, and data-access surfaces; the dashboard (67 files) was sampled at the load-bearing pages and the auth/data boundary; dependencies were assessed from manifests plus `bun.lock`. 11 specialist auditors (security split into authz / injection / secrets) produced 54 findings; all 8 High findings were re-checked by an independent adversarial pass.
- **Exclusions**: `node_modules/`, `apps/backend/convex/_generated/`, `apps/dashboard/.svelte-kit/` and `build/`, `packages/sdk/dist/`, `bun.lock`, `CHANGELOG.md`, `infra/.env`, `.git/`, `.changeset/`, `.vscode/`, `.claude/`.

## Overall score

**75 / 100 - Grade C (adequate, real gaps).**

A genuinely well-engineered pre-1.0 self-hosted tracker whose load-bearing parts clear the bar: multi-tenant authorization is centralized and applied without a single observed gap, the Sentry wire-compat core is a pure, exemplary unit-tested package, egress is a single SSRF chokepoint, and the forward-facing docs are unusually honest. It is held to a C by two confirmed High input-safety gaps on the internet-facing boundaries (an uptime probe that bypasses the SSRF guard, and a decompression-bomb guard that runs only after full inflation with no body cap), a zero-test Convex backend and dashboard where every isolation and auth guard ships unverified, and a systemic failure-visibility hole where cron-driven alerts mark themselves "fired" even when delivery dropped. No surviving Critical, so nothing is hard-capped; the volume of confirmed High systemic findings on the highest-risk surfaces is what keeps it out of B range.

*Calibration*: graded against the bar for a security-sensitive, multi-tenant, self-hosted ingestion service approaching 1.0, not a finished Sentry SaaS and not a weekend prototype. Multi-tenant isolation, the two internet-facing trust boundaries (DSN ingest, SSRF egress), and doc-vs-code honesty on "Done" claims are held to a high bar; deliberately Partial/Planned features disclosed in the parity matrix are not penalized.

## Systemic patterns (root causes)

- **The SSRF chokepoint is mandatory, but one user-influenced outbound path bypasses it.** Members: `SEC-inject-001`, `ERR-005`. Root fix: route every user-influenced outbound request through `safeFetch` (literal guard + DoH rebinding check, re-validated per redirect hop); fix the uptime probe now and treat "no raw `fetch` on a user-supplied URL" as an invariant grep-checked in CI.
- **No request-body cap and post-hoc-only decompression guard on the internet-facing DSN-authed write endpoints.** Members: `SEC-inject-002`, `SEC-secrets-001`, `ERR-003`. Root fix: cap compressed body size before buffering and enforce the decompressed budget incrementally (fflate streaming with an abort, or a fixed `out` buffer); apply the same cap to artifact upload's storage fallback.
- **Cron-driven side effects swallow failures with no record and no log, so the monitoring product cannot self-monitor.** Members: `ERR-001`, `OBS-002`, `OBS-004`. Root fix: extend the `alertDeliveries`-style per-attempt outcome record (and at minimum a log line) to metric/usage/uptime/tracker delivery and to cron heartbeats; decouple the "fired" marker from delivery success and surface the records in the dashboard.
- **Hand-maintained per-table tenant lists drift silently across purge, restamp, and retention.** Members: `QUAL-001`. Root fix: declare one registry of tenant-scoped tables (with `projectId`-prefixed index and org-stamp flag) and drive purge, restamp, and retention from it with a compile-time exhaustiveness check against the schema; add the missing `webhooks`/`webhookDeliveries` coverage.
- **Security-critical wiring is entirely untested while the pure protocol layer is exemplary.** Members: `TEST-001`, `TEST-002`, `TEST-003`. Root fix: add a Convex/dashboard/SDK test harness asserting the isolation, auth, rate-limit, and scope guards, and wire `bun run test` (`--filter '*' --if-present`) into CI so new package tests gate automatically.
- **Incomplete post-migration cleanup leaves dead deps and self-contradicting docs about the Convex-only architecture.** Members: `DOC-001`, `DOC-002`, `QUAL-003`, `QUAL-006`, `DEP-003`. Root fix: finish the Convex-only sweep: remove `pg`/`jose` from the catalog and dependabot, fix the schema docblock and README "backed by Postgres" framing, and rewrite the stale `ARCHITECTURE.md` sections.

### [ERR-001] ~~Cron-driven metric and usage alert delivery failures are silently swallowed and never recorded~~ [RESOLVED - Slice 3]
- Status: RESOLVED (Slice 3). metric/usage alert crons now capture per-channel ok/detail, write a `notificationDeliveries` row per attempt, log failures, and advance the fired marker (lastFiredAt / lastFiredMonth) only when at least one channel delivered. Mirrors the issue-alert `alerts.ts` pattern.
- Severity: High | Confidence: Confirmed | Effort: M | Dimension: Error Handling & Resilience
- Location: `apps/backend/convex/metricAlerts.ts:234-263`, `apps/backend/convex/usageAlerts.ts:146-170`
- Evidence: In evaluateMetricAlerts the per-channel send is wrapped in `try { ... } catch { /* Best-effort delivery; do not block other channels/alerts. */ }` (metricAlerts.ts:235-256) with no record of success or failure. recordMetricFiring (258-262) then sets lastFiredAt UNCONDITIONALLY, even when every channel threw or the webhook returned non-2xx (safeFetch's Response is never inspected for `.ok`). Because the next evaluation suppresses re-firing while `now - lastFiredAt < windowMinutes*60_000` (line 225), one failed delivery marks the alert 'fired' and silences it for the whole window. usageAlerts.ts:146-170 is the identical pattern (fires at most once per month: a single swallowed failure suppresses the alert for the rest of the calendar month). Unlike alerts.ts, there is no alertDeliveries-equivalent row, so an operator querying the DB has zero signal a metric/quota alert ever failed.
- Impact: A misconfigured or transiently-down Slack/webhook/SMTP endpoint causes metric-threshold alerts (p95 latency, error count, crash-free rate) and quota-usage alerts to be marked delivered and go silent for the rest of the window/month, with no visible trace anywhere. The operator believes alerting works; a real latency spike or crash-free regression produces no notification and no failure record to diagnose. This is the central observability gap for the entire scheduled-alert subsystem.
- Recommendation: Mirror the ingest-path discipline: inspect the outcome (check `res.ok` from safeFetch and `res.ok`/skipped from sendEmail), persist a per-channel delivery record (success/failure + detail) to a table the dashboard can surface, and only advance lastFiredAt/lastFiredMonth when at least one channel actually succeeded (or record the firing but separately track that delivery failed so it retries next tick).
- Verify the fix: Configure a metric alert with a webhook pointing at a URL that returns 500; trigger the threshold; confirm lastFiredAt is NOT advanced (or a failure row is written) and the next cron tick re-attempts. Repeat with a usage alert and a dead SMTP host.
- Verification: Adversarially verified; holds at High/Confirmed. metricAlerts.ts:258 and usageAlerts.ts:166 mark the alert fired unconditionally after a bare-catch channel loop, and safeFetch returns 4xx/5xx as a resolved Response the cron never inspects. The ingest-path alerts.ts already records per-attempt ok/detail into alertDeliveries and only dedupes on ok; mirror that pattern.
- Related: Contrast with alerts.ts:243-273 recordDelivery, which does this correctly.

### [ERR-002] ~~Ingest batch persistence is non-atomic with no batch-level idempotency, so a mid-batch failure or HTTP-action retry leaves partial/duplicated state~~ [RESOLVED - Slice 5]
- Status: RESOLVED (Slice 5). `recordEvent`/`recordTransaction` return whether they inserted, and usage now counts only actually-inserted rows, so a full-batch retry no longer double-counts toward the quota (live-verified: a resent event/transaction reports duplicate/`inserted:false`). Replay/attachment/feedback writes dedupe on a stable id; a deduped or failed write drops the already-stored blob (also closes ERR-004). Mid-batch failures no longer over-count usage because the count derives from inserts, not payload length.
- Severity: High | Confidence: Confirmed | Effort: L | Dimension: Error Handling & Resilience
- Location: `apps/backend/convex/ingest.ts:275-485`, `apps/backend/convex/ingest.ts:476-485`
- Evidence: The ingest httpAction persists each envelope item as a SEPARATE ctx.runMutation transaction in sequence: events (281), transactions (306), sessions (330), checkins (359), replays (388, after ctx.storage.store at 384), profiles (405), attachments (428, after store at 423), feedback (443/459), and finally recordUsage (477) ONCE at the end. None of this is wrapped in error handling. If any mutation throws partway through (e.g. a transient Convex write conflict on the issues upsert at recordEvent, or a storage.store failure), the action returns 5xx with earlier items already committed and later items (including the single recordUsage write that backs quota/spike accounting) never run. Sentry SDKs retry on 5xx, and the HTTP action has no batch-level dedupe key, so the retry re-runs the whole batch. Per-event recordEvent/recordTransaction are individually idempotent on (projectId,eventId) (549-555, 513-519), but sessions (recordSession), checkins, replays (storage blob + recordReplaySegment), attachments (a fresh storage blob + recordAttachment each time), feedback, and crucially recordUsage are NOT eventId-idempotent, so a retry double-counts usage and creates duplicate attachment/replay blobs and feedback rows.
- Impact: On any partial failure: quota/spike counters drift (recordUsage may never run, undercounting; or run twice on retry, overcounting and prematurely throttling a tenant), duplicate attachment/replay blobs accumulate in storage, and duplicate feedback rows appear. Because recordUsage is the last write, the common 'late mutation throws' case silently loses an entire batch's usage accounting while the data is stored, breaking the documented monthly-quota enforcement.
- Recommendation: Make recordUsage idempotent per batch (key it on the envelope event_id or a request-derived token), or move usage accounting into the same transaction as the first persisted item; make attachment/replay/feedback inserts idempotent on a natural key; and wrap the per-item loops so a single item failure does not abort the batch and trigger a full retry. At minimum, dedupe the batch on env.header.event_id before any side effects.
- Verify the fix: Send an envelope containing an event + attachment + client_report twice (simulating an SDK retry) and confirm usageDaily increments once and only one attachment blob exists. Inject a throw in recordSession and confirm earlier events plus recordUsage are not left inconsistent.
- Verification: Adversarially verified; holds at High. Each envelope item is a separate non-retried transaction ending in one recordUsage write, and only events/transactions dedupe on eventId, so a full-batch SDK retry double-counts usage (driving false quota throttling) and duplicates attachment/replay/feedback rows and blobs. Evidence correction: the dominant trigger is not an OCC write-conflict (Convex auto-retries those) but ctx.storage.store failures, action timeouts, and non-OCC errors producing partial commits.

### [SEC-inject-003] redirectTo from query string passed to goto() without same-origin validation
- Severity: Low | Confidence: Likely | Effort: S | Dimension: Security -- Injection, SSRF, Input Safety
- Location: `apps/dashboard/src/routes/login/+page.svelte:30`, `apps/dashboard/src/routes/signup/+page.svelte:33`, `apps/dashboard/src/routes/invite/[token]/+page.svelte:42`
- Evidence: login and signup read `redirectTo` straight from page.url.searchParams and pass it to `goto(...)` after authentication with no allowlist/relative-path check (login:30, signup:33). A crafted link like /login?redirectTo=//evil.example or /login?redirectTo=https://evil.example reflects an attacker-controlled destination into the post-login navigation. invite/[token] also does `location.href = /login?redirectTo=/invite/${token}` (invite:42), but there the value is a fixed internal path. The practical blast radius is limited because SvelteKit's goto() resolves the target against the app origin and refuses cross-origin navigation (it throws rather than navigating to a different origin), so a fully external https://evil.example does not redirect; a protocol-relative //evil.example likewise resolves cross-origin and is rejected. The residual risk is a confusing same-origin path redirect and reliance on framework behavior rather than an explicit guard.
- Impact: At worst a same-origin redirect to an unexpected internal path; classic open-redirect phishing to an external site is blocked by goto()'s same-origin enforcement, so impact is low. Still a latent issue if a future change swaps goto() for window.location.assign(redirectTo) (which would NOT enforce same-origin).
- Recommendation: Validate redirectTo before navigating: accept only values that start with a single '/' and not '//' or '/\\' (i.e. a same-origin absolute path), else fall back to /dashboard. A 3-line guard makes the safety explicit instead of depending on goto() internals.
- Verify the fix: Visit /login?redirectTo=//evil.example and /login?redirectTo=https://evil.example, sign in, and confirm navigation lands on /dashboard (or a sanitized path), not the external host. Add a unit/e2e assertion on the sanitizer.
- Related: Apply the same path-only guard anywhere a query param feeds navigation or location.* assignment.
