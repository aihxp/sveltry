### [SEC-inject-001] ~~Stored XSS: commit `url` rendered into an href with no scheme validation (javascript: executes)~~ [RESOLVED - RA-Slice 1]
- Status: RESOLVED (RA-Slice 1). Added a shared `httpUrlOnly` helper (`@sveltry/protocol`, unit-tested) and apply it before storing the commit URL (`commits.ts`) and the tracker URL (`integrations.recordTrackerLink`), so a `javascript:`/`data:` scheme is dropped to `undefined`. A defense-in-depth `safeHref()` (dashboard utils) wraps the two render sinks (commit + tracker href). No other raw `.url` href bindings remain.
- Severity: High | Confidence: Confirmed | Effort: S | Dimension: Security -- Injection, SSRF, Input Safety
- Location: `apps/backend/convex/commits.ts:76`, `apps/backend/convex/commits.ts:233`, `apps/dashboard/src/routes/(app)/issues/[id]/+page.svelte:323-329`
- Evidence: The set-commits endpoint stores the per-commit `url` field verbatim with only a typeof-string check: `url: typeof c.url === 'string' ? c.url : undefined` (commits.ts:76). suspectCommitsForIssue surfaces it unchanged (`url: c.url ?? null`, commits.ts:233). The issue-detail page binds it straight into an anchor: `<a href={c.url} target="_blank" rel="noreferrer">` (issues/[id]/+page.svelte:325). No scheme allowlist runs at any hop, so a `javascript:alert(document.cookie)` (or `data:`) URL is stored and rendered; clicking the suspect-commit link executes script in the operator's authenticated dashboard origin. The write path is DSN-key authenticated (resolveDsnRequest), so this is reachable by anyone holding a valid project DSN key (every SDK config ships one) or a leaked key, not by an arbitrary anonymous request. The same class also applies to `i.trackerUrl` (issues/[id]/+page.svelte:275), which is taken from an external tracker's JSON response (parseLinearResult/parseJiraResult) with no scheme check. The contrast is what makes this a clear miss, not a judgment call: the repoConfig.baseUrl field IS validated to be https: server-side (projects.ts:293), so the codebase already knows the pattern; it was just not applied to commit/tracker URLs.
- Impact: Stored XSS in the authenticated dashboard origin. An attacker with a project DSN key (or a malicious/compromised tracker integration for the trackerUrl variant) can land script that runs with the victim operator's session, enabling org-data exfiltration, issue tampering, or pivoting via the operator's privileges. Severity is High rather than Critical because (a) the injection requires a valid DSN key rather than being anonymous, and (b) execution requires an operator to click the rendered link.
- Recommendation: Validate the URL scheme to http/https before storing (or before rendering). Cheapest robust fix: in commits.ts, after the typeof check, parse with `new URL(c.url)` and keep it only if `protocol === 'https:' || protocol === 'http:'`, else drop to undefined; do the same for outcome.url in integrations.recordTrackerLink. A defense-in-depth client guard (a tiny safeHref(url) helper that returns '#' for non-http(s) schemes, applied at all three href bindings) closes it even for already-stored rows. Reuse the exact pattern already in projects.ts:288-295.
- Verify the fix: Upload a commit via POST /releases/commits with `url: "javascript:alert(1)"` for a valid DSN; open the linked issue's Suspect Commits card and confirm the anchor href is sanitized (not `javascript:`). Add a unit/integration test asserting a non-http(s) commit url is stored as undefined (or rendered as a safe href). Repeat for a tracker that returns a javascript: url.
- Verification: Adversarially verified; holds at High/Confirmed. Refutation at the framework, CSP, and validation layers all failed: Svelte 5.56.3 does not sanitize a dynamic `href` value, no `isSafeUrl`/protocol-allowlist utility exists in the dashboard, and SvelteKit's default CSP adds no script-src backstop. The contrast is load-bearing and accurate: `repoConfig.baseUrl` IS server-side https-validated at `projects.ts:289-295` and flows into an href, proving the pattern exists in-repo but was not applied to commit/tracker URLs (the tracker variant via `parseLinearResult` -> `recordTrackerLink` -> `+page.svelte:275` is confirmed too). High not Critical: the write needs a valid (authenticated) DSN key and execution needs an operator to click the link in their dashboard origin.
- Related: Any user/SDK/3rd-party-supplied string rendered into an href: commit url, trackerUrl, release url (publicApi releaseView r.url, deployView d.url) -- audit each for scheme validation.

### [ARC-001] ~~Tenant-table registry is documented as the single source of truth but its lifecycle consumers are hand-maintained with no compile-time link to it~~ [RESOLVED - RA-Slice 4]
- Status: RESOLVED (RA-Slice 4) via recommendation (a), the real fix. `purgeProjectData` and `restampProjectOrg` are now driven by `Record<ProjectScopedTable, ...>` drainer maps (`PROJECT_PURGERS` / `ORG_DRAINERS`) and iterate `TENANT_SCOPED_TABLES`. Because the map key type IS the registry's element type, TypeScript fails the build if a registered table has no purge / transfer step, so the schema -> registry -> drainer chain is now compile-enforced end to end (adding a `projectId` table is a compile error until it is registered AND drained by both ops). `projectKeys` (re-stamped atomically in `transferProject`) and `spikeWindows` (no `organizationId`) are explicit no-op drainers so the registry stays exhaustive. The `tenantTables.ts` docblock was corrected to note the retention sweep is an intentional high-volume subset, not registry-driven. Behavior preserved (verified by the lifecycle smoke tests in RA-Slice 4 and the exhaustive seed-all-32-tables `lifecycle.test.ts` in RA-Slice 5, TEST-001).
- Severity: Medium | Confidence: Confirmed | Effort: M | Dimension: Architecture & Design
- Location: `apps/backend/convex/lib/tenantTables.ts:3-70`, `apps/backend/convex/projectLifecycle.ts:64-348`, `apps/backend/convex/projectLifecycle.ts:366-657`, `apps/backend/convex/maintenance.ts:22-78`, `docs/ARCHITECTURE.md:112-113`
- Evidence: tenantTables.ts:3-17 states the registry is the 'single source of truth' that delete (purgeProjectData), transfer (restampProjectOrg), and the retention sweep 'must each cover,' and frames the compile-time assertion as why 'the registry cannot silently fall behind.' But that assertion (lines 60-69) only proves registry == schema. purgeProjectData (projectLifecycle.ts:64-348) and restampProjectOrg (366-657) never import or iterate TENANT_SCOPED_TABLES; they are hand-written sequences of purgeRows('events')/drainer closures, and sweepRetention (maintenance.ts:33-78) covers only 3 tables by hand. Coverage happens to be correct today (the purge region queries exactly the 32 registered tables; restamp covers the 30 org-bearing/detach tables), but if a future table gains a projectId, the build forces a registry edit yet stays green even if the developer forgets the matching purgeRows call or drainer, re-introducing the exact webhooks-orphaning drift the registry comment says it prevents. Convex's typed per-table index API makes a fully data-driven loop non-trivial, which is why the lists are hand-written, but a registry-keyed exhaustiveness check (a Record<ProjectScopedTable, drainer> the compiler requires to be total) would close the gap.
- Impact: The safety property is weaker than the code and docs advertise. The registry catches schema drift but not lifecycle drift, so a future schema addition can silently leave purge/restamp incomplete, orphaning tenant rows on delete (including a plaintext webhook secret, the historic bug) or leaving them stamped with the source org's id on transfer (cross-tenant isolation residue). It also misleads a maintainer who trusts the 'single source of truth' framing into thinking the build guards them when it does not.
- Recommendation: Tie the lifecycle functions to the registry so the compiler enforces completeness. Either (a) restructure purge/restamp to iterate a Record<ProjectScopedTable, (ctx,pid)=>...> map that TypeScript requires to be exhaustive over ProjectScopedTable (forcing a compile error when a registered table has no drainer), or (b) at minimum soften the tenantTables.ts and ARCHITECTURE.md wording to state explicitly that the registry guards only the schema side and lifecycle coverage is hand-verified, and add a test asserting the purge query set equals TENANT_SCOPED_TABLES.
- Verify the fix: Add a new project-scoped table with a projectId to schema.ts and confirm the build now fails (or a test fails) until purgeProjectData and restampProjectOrg are updated, not only until tenantTables.ts is updated.
- Related: Pairs with any lifecycle-coverage finding from other lenses (purge/restamp/retention table omissions).

### [SEC-secrets-002] API token authentication uses a plain DB-indexed hash equality lookup (not constant-time), a deviation from timing-safe token comparison
- Severity: Low | Confidence: Likely | Effort: M | Dimension: Security (secrets, crypto, exposure)
- Location: `apps/backend/convex/apiTokens.ts:93-105 (resolveApiToken, withIndex by_hash eq sha1Hex(rawToken))`
- Evidence: resolveApiToken hashes the presented bearer token with sha1Hex and looks it up via the by_hash index (q.eq('tokenHash', ...)). A B-tree index equality comparison is not a constant-time compare, so in principle the lookup leaks timing about how far the stored hash prefixes match a candidate. In practice this is heavily mitigated: the comparison is against the SHA-1 *digest* (not the secret), the secret carries 256 bits of entropy, and a TOKEN_RE shape pre-check (apiTokens.ts:95) gates the lookup, making a timing-recovery attack against a hashed, high-entropy value impractical. It is recorded only as a deviation from the timing-safe-compare convention, not a practical break.
- Impact: Theoretical timing side channel on token lookup. Not practically exploitable given the digest-not-secret comparison and 256-bit entropy; included as a hardening observation, not an active vulnerability.
- Recommendation: Accept as-is for this threat model (this is the standard hashed-token-index pattern and is fine for high-entropy random tokens), or, if maximal rigor is wanted, document explicitly that lookup is over a digest of a 256-bit secret so timing recovery is infeasible. No code change is required for the stated calibration.
- Verify the fix: Confirm the comment at apiTokens.ts:96-98 (entropy + preimage-resistance rationale) is retained; optionally add a note that the index lookup is intentionally not constant-time because the compared value is a digest of a 256-bit secret.
- Related: none
